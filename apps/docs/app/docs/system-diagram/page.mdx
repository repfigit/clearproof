# System Diagram

## The core flow

```mermaid
graph LR
    A["🏦 Originating<br/>VASP"] -->|"ZK proof +<br/>encrypted PII"| B["🏦 Beneficiary<br/>VASP"]
    B -->|"optional"| C["⛓️ On-chain<br/>verification"]

    click A "/docs#terminology" "What is a VASP?"
    click C "/docs/contracts" "Smart Contracts"

    style A fill:#1e1b4b,stroke:#6366f1,color:#e2e8f0
    style B fill:#1e1b4b,stroke:#06b6d4,color:#e2e8f0
    style C fill:#0f172a,stroke:#22d3ee,color:#e2e8f0
```

The originating [VASP](/docs#terminology) generates a [ZK proof](/docs/circuits) that compliance was performed, encrypts the PII, and sends both to the counterparty. The counterparty verifies the proof without seeing the raw data. [On-chain recording](/docs/contracts) is optional.

## Step by step

### 1. Proof generation (originating VASP)

```mermaid
graph TD
    Req["Transfer request"] --> Cred["Look up<br/>zkKYC credential"]
    Req --> Sanc["Build sanctions<br/>gap proof"]
    Req --> Iss["Build issuer<br/>membership proof"]

    Cred --> Circuit["Circom circuit<br/>(Groth16, ~500ms)"]
    Sanc --> Circuit
    Iss --> Circuit

    Circuit --> Proof["192-byte proof<br/>+ 16 public signals"]

    click Cred "/docs/circuits#credential-validity" "Credential validity circuit"
    click Sanc "/docs/circuits#sanctions-non-membership" "Sanctions non-membership circuit"
    click Iss "/docs/circuits#credential-validity" "Issuer membership proof"
    click Circuit "/docs/circuits" "Circuit architecture"
    click Proof "/docs/circuits#public-signals" "16 public signals"

    style Req fill:#1e293b,stroke:#94a3b8,color:#e2e8f0
    style Circuit fill:#1e1b4b,stroke:#6366f1,color:#e2e8f0
    style Proof fill:#065f46,stroke:#10b981,color:#ecfdf5
```

The [Circom circuit](/docs/circuits) composes four subcircuits: [credential validity](/docs/circuits#credential-validity), [sanctions non-membership](/docs/circuits#sanctions-non-membership), [amount tier](/docs/circuits#amount-tier), and domain binding + expiration. Output is a 192-byte Groth16 proof with [16 public signals](/docs/circuits#public-signals).

### 2. What goes into the hybrid payload

```mermaid
graph LR
    subgraph Payload["Hybrid Payload"]
        direction TB
        ZK["ZK Proof<br/>192 bytes<br/>(publicly verifiable)"]
        PII["Encrypted PII<br/>AES-256-GCM<br/>(only counterparty<br/>can decrypt)"]
        Meta["Metadata<br/>proof_id, expires_at,<br/>vasp_did"]
    end

    click ZK "/docs/circuits" "Circuit architecture"
    click PII "/docs/security#pii-protection" "PII encryption details"

    style Payload fill:#0f172a,stroke:#6366f1,color:#e2e8f0
    style ZK fill:#1e1b4b,stroke:#6366f1,color:#e2e8f0
    style PII fill:#1e1b4b,stroke:#06b6d4,color:#e2e8f0
    style Meta fill:#1e293b,stroke:#94a3b8,color:#e2e8f0
```

The [hybrid payload](/docs/architecture#hybrid-payload) bundles the [ZK proof](/docs/circuits) with [AES-256-GCM encrypted PII](/docs/security#pii-protection). The proof is publicly verifiable. The PII is readable only by the intended counterparty.

The payload is transmitted via existing Travel Rule protocols. clearproof replaces the payload content, not the transport.

### 4. Beneficiary verification

```mermaid

    Receive["Receive<br/>payload"] --> Verify["Verify ZK<br/>proof (<1ms)"]
    Verify --> Decrypt["Decrypt PII<br/>for records"]
    Decrypt --> Store["Store 5 years<br/>(FATF requirement)"]

    click Verify "/docs/sdk" "TypeScript SDK — verifyProof()"
    click Decrypt "/docs/security#pii-protection" "PII decryption"
    click Store "/docs/security" "Security & retention"

    style Receive fill:#1e293b,stroke:#94a3b8,color:#e2e8f0
    style Verify fill:#065f46,stroke:#10b981,color:#ecfdf5
    style Decrypt fill:#1e1b4b,stroke:#06b6d4,color:#e2e8f0
    style Store fill:#1e293b,stroke:#94a3b8,color:#e2e8f0
```

The beneficiary [verifies the proof](/docs/sdk) locally in under 1ms using the [TypeScript SDK](/docs/sdk). PII is [decrypted](/docs/security#pii-protection) for record-keeping as required by FATF.

## Regulatory audit path

```mermaid
graph LR
    Reg["Regulator<br/>issues subpoena"] --> VASP["VASP retrieves<br/>proof + encrypted PII<br/>from records"]
    VASP --> Decrypt["Decrypt PII via<br/>HKDF key re-derivation"]
    Decrypt --> Produce["Produce plaintext<br/>IVMS101 data +<br/>on-chain audit trail"]

    click VASP "/docs/security#pii-protection" "PII encryption & key management"
    click Produce "/docs/security" "Security architecture"

    style Reg fill:#1e293b,stroke:#94a3b8,color:#e2e8f0
    style VASP fill:#1e1b4b,stroke:#6366f1,color:#e2e8f0
    style Decrypt fill:#1e1b4b,stroke:#06b6d4,color:#e2e8f0
    style Produce fill:#065f46,stroke:#10b981,color:#ecfdf5
```

Regulators access plaintext PII through the [VASP's internal records](/docs/security#pii-protection), not through the blockchain. The ZK proof and [on-chain events](/docs/contracts#complianceregistry) provide a verifiable audit trail.
